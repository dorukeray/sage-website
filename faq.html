Answers

  • Why should I use Sage?

    It depends on your use case, but in general, Sage has a few key features that stand out. For example, Sage enables you to...

    • Query for only what you want, and get exact, predictable results.
    • Create a representation of your data that feels natural (a graph).
    • Describe and manage the data requirements of your app.
    • Ensure that all of your data is flexibly typed with a schema.
    • Reduce the need for breaking changes, but utilize a built-in mechanism for deprecations when you need to.

    Our homepage outlines even more reasons to use Sage.

  • Does Sage replace REST or GraphQL?

    No, not necessarily. They all handle APIs and can serve similar purposes from a business perspective. Sage is can be considered as a minimalist alternative to GraphQL, but this wouldn't be a definitive replacement.

    Sage and REST can actually co-exist in your stack. For example, you can abstract REST APIs behind a Sage service.

  • How can I learn Sage?

    This website has the official walkthrough and many other resources available to help you learn Sage. In our documentation, you can find the best practices and other articles that explain essential concepts and how they work.

    Before you start your learning journey, make sure you know what an API is and how communication generally works between client and server.

    • General

  • Is Sage frontend or backend?

    Both. Sage specifies how you can exchange information between client and server. This includes...

    • how the server can indicate what data is available,
    • how the client should format requests,
    • how the server should execute these queries,
    • what the client will receive in response.

  • Does Sage use HTTP?

    Yes, Sage is typically served over HTTP. This is largely due to how pervasive the HTTP protocol is in our industry. But it helps that you can try out Sage by creating a single HTTP request. Guidelines for setting up a Sage service to operate over HTTP are available in our documentation.

    While HTTP is the most common choice for client-server protocol, it’s not the only one. Sage is agnostic to the transport layer. So, for example, you could use WebSockets for communicating with a Sage service, instead of HTTP to consume realtime data.

  • What kind of protocol is Sage?

    Sage is an application-layer protocol and does not require or dictate a particular transport. It is a standard about how clients & services interact and exchange data, in a query-based and entity-focused way.

  • How does Sage affect my product’s performance?

    On a typical Sage service, every artifact (attribute, act or link) on every type has a focused, single-purpose function for resolving that value. As a result, there are some inherent performance benefits. Minimizing over-fetching and making fewer roundtrips to the server are two of them.

    Other performance factors should be considered when building out your Sage implementation. For example, it’s possible for a Sage service to be ‘chatty’ and repeatedly load data from your database. This is commonly solved by implementing a batching technique, like GraphQL's DataLoader does.

  • What is a Sage client and why would I use one?

    Sage clients are necessary for...

    • handling queries
    • networking
    • describing data requirements
    • managing the data locally

    They use the underlying capabilities and the structure of a Sage API to handle these; in the end which simplifies your data interactions.

  • Does Sage replace databases or ORMs?

    No, Sage is a protocol typically used for remote client-server data interactions. It's agnostic to the data source(s) used and doesn’t implement an object-relational mapping technique. But it provides a way for describing your data schema as an entity graph.

  • Who is behind Sage?

    The protocol is originally created by Doruk Eray at Dorkodu. We'll seperate the Sage protocol completely from Dorkodu, once the wide adoption is achieved and outsider contributors start involving in the project.

    The Sage protocol and all related projects are open source, so anyone is welcome to contribute. That said, there is a structure in place behind the repositories. This exists to resolve conflicts within the community and guiding technical decisions.

    • Best Practices

  • Is Sage scalable?

    Yes, Sage is designed to be scalable and used in production under a very high load.

    Sage comes with some built-in performance boosts that can help. But once you push it to production, you're responsible for scaling it across instances and monitoring performance.

  • What are the security concerns with Sage?

    Most of the security concerns associated with Sage are typical for any API or service. A few examples: SQL injections, Denial of Service (DoS) attacks, or someone abusing flawed authentication.

    No matter the concern, it’s important to be proactive. There are many ways to securing your Sage server. Using a timeout or throttling queries based on the server time it needs to complete are some potential approaches.

  • How does auth* work with Sage?

    We recommend enforcing authorization behavior in the business logic layer. That way, you have a single source of truth for authorization. For a more detailed explanation, go to our Auth* documentation.

    You can implement authentication with common patterns, such as OAuth or JWT. There’s nothing special about authentication within the Sage protocol. Also, if you’re working with a pipeline model, we recommend that Sage should be placed after all auth* middleware.

  • How does versioning work in Sage?

    There’s nothing that will prevent a Sage service from being versioned like any other API. That said, Sage avoids versioning by design.

    Instead, Sage helps you continually evolve your schema. For example, Sage only returns the data that’s explicitly requested. This means that you can add new features (and all the associated types and artifacts) without creating a breaking change or bloating results for existing queries.

    You can read more about how versioning works in Sage in that specific Best Practices section.

  • How can I document my Sage API?

    One of the benefits of Sage is that it's inherently self-documenting. This means that you’re able to explore what data is exposed by your Sage API. This includes the artifacts, types, and more.

    You can also add a description to provide supplementary notes about your endpoint.

    For many, this provides enough API reference documentation. But it doesn’t reduce the need for other forms of documentation. You'll likely still need to create guides that explain how the general concepts tie into your specific use case.

    • Protocol

  • What is the best way to follow protocol releases?

    The latest working draft of the Sage protocol whitepaper can be found on the GitHub repo (dorkodu/sage). All editions are also available at permalinks that match their release tag.

    The entire process behind each release is open source. You can monitor specification proposals and the progress by following pull requests and commits in the repository.

  • How can I contribute to the Sage protocol?

    Sage is still evolving and contributions are very welcome! The specification (including the latest working draft) is open source. Contributor guidelines are available on GitHub.

    There are more ways to get involved with Sage beyond the specification though:

    • Implementing Sage in your favorite language.
    • Contributing to existing projects for Sage. (e.g. reference implementations, tools, educational content etc.)
    • Updating the content on this website and the documentation.
    • Sponsor our work.
